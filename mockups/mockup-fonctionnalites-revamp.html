Logique du revamp

  • 1. Renommage — « Feature flags » (jargon dev) → « Fonctionnalités » (compréhensible immédiatement par un super-admin métier).
  • 2. KPI strip 4 cards — Total / Actives / Personnalisées / Désactivées avec accent visuel sur les overrides (gold) pour identifier vite l'exception.
  • 3. Recherche + filtres rapides — pour 14+ flags, indispensable. Filter as you type sur nom/slug/description.
  • 4. Cartes plutôt que lignes — chaque fonctionnalité respire : icône, nom convivial, description, slug technique discret, badge maturité (Stable/Beta/Nouveau).
  • 5. Toggle simple par défaut — un clic active/désactive (override). Plus de segmented control 3-options en première lecture.
  • 6. Override visuel fort — bord coloré gauche (rouge/vert) + bandeau motif + auteur + date. L'admin voit immédiatement ce qui dévie du global.
  • 7. Catégories — IA & Productivité, Reporting, Collaboration, Sécurité… pour scanner par usage métier plutôt que par ordre alphabétique de slug.
  • 8. Maturité signalée — badges Beta/Nouveau/Stable pour décider si on peut activer pour un cabinet client en prod.
  • 9. Action "Réinitialiser" — un seul lien clair pour repasser en mode hérité, plutôt que 3 boutons confus.
  • 10. Historique des changements — collapsible accessible mais pas encombrant, pour audit complet sans noyer la lecture.

Prérequis backend

  • Catégorisation — ajouter un champ category sur feature_flags (ai, reporting, collab, security, beta).
  • Maturité — ajouter un champ maturity (stable, beta, new) ou utiliser un tag.
  • Description / icône — description existe déjà, on peut ajouter icon_name (lucide-react slug) optionnel.
  • Historique — déjà présent dans audit_log filtrable par target_id = flag_id.
