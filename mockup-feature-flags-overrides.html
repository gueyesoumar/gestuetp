Onglet « Feature flags » dans la fiche cabinet

Chaque cabinet voit la liste complète des flags avec 3 états : Hérite du global, Forcé ON, ou Forcé OFF. Toute modification exige un motif texte et apparaît dans l'audit log. La résolution se fait à 2 niveaux côté useFeatureFlag() : override cabinet d'abord, fallback global ensuite.

04

Architecture · résolution à 2 niveaux dans useFeatureFlag()

Le code consommateur (useFeatureFlag('slug')) ne change pas. La résolution interne devient à 2 niveaux : override de l'org du user d'abord, fallback sur is_globally_enabled ensuite.
-- Migration 00073 CREATE TABLE public.feature_flag_overrides ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), flag_id uuid NOT NULL REFERENCES public.feature_flags(id) ON DELETE CASCADE, organization_id uuid NOT NULL REFERENCES public.organizations(id) ON DELETE CASCADE, enabled boolean NOT NULL, reason text NOT NULL CHECK (length(trim(reason)) > 0), updated_by uuid REFERENCES public.users(id) ON DELETE SET NULL, updated_at timestamptz NOT NULL DEFAULT now(), UNIQUE (flag_id, organization_id) ); // src/hooks/useFeatureFlag.ts (résolution à 2 niveaux) export function useFeatureFlag(slug: string) { const { profile } = useAuth() // 1. override pour l'org du user courant ? const override = useOverride(slug, profile?.organization_id) if (override !== null) return { enabled: override.enabled, source: 'override' } // 2. fallback global return useGlobal(slug) // is_globally_enabled }

Cas d'usage opérationnel

Soft-launch report_generator_advanced sur 1 cabinet pilote → laisser global OFF → activer pour tous quand prêt.

Cas d'usage support

Le client se plaint que smart_analyse_control renvoie des erreurs → on force OFF pour lui le temps du fix → ses concurrents continuent normalement.
