Espace super-admin · Mockup d'impact

Gëstu Admin — la console côté éditeur, séparée de l'app cabinet

Une sous-application /admin gardée par un flag is_platform_owner sur users, avec sa propre identité visuelle (sidebar nuit, mode-bar or « ADMIN MODE »). On voit tous les cabinets, on suit les KPI plateforme, on intervient sur les comptes en support — tout en gardant une trace auditable de chaque geste. Le mockup propose un découpage Phase 1 (immédiat, opérationnel) / Phase 2 (extension), à arbitrer ensemble.

06

Impersonation — voir l'app comme un utilisateur

Phase 2
Pour reproduire un bug ou faire du support, le super-admin endosse l'identité d'un utilisateur sur une session limitée (30 min, max). Une bande or persistante rappelle l'état pour éviter toute confusion. Action loguée systématiquement.

08

Schéma technique · modèle minimal

Pas besoin d'un nouveau service : un flag is_platform_owner sur la table users + un audit log dédié + un point d'extension pour les flags. Le reste réutilise les permissions service-role.
-- Migration 00067_platform_owner_flag_up.sql ALTER TABLE public.users ADD COLUMN is_platform_owner boolean NOT NULL DEFAULT false; -- Helper SECURITY DEFINER pour les RLS qui veulent permettre l'accès super-admin CREATE OR REPLACE FUNCTION public.is_platform_owner() RETURNS boolean LANGUAGE sql SECURITY DEFINER STABLE AS $$ SELECT EXISTS ( SELECT 1 FROM public.users WHERE auth_id = auth.uid() AND is_platform_owner = true ); $$; -- Migration 00068_admin_audit_log_up.sql CREATE TABLE public.admin_audit_log ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), actor_id uuid NOT NULL REFERENCES public.users(id), action text NOT NULL, -- 'suspend_cabinet', 'reset_password', 'impersonate', etc. target_type text NOT NULL, -- 'cabinet', 'user', 'mission' target_id uuid, reason text NOT NULL, -- motif obligatoire metadata jsonb, created_at timestamptz NOT NULL DEFAULT now() ); -- RLS : lecture réservée aux platform_owners, insert via Edge Function uniquement ALTER TABLE public.admin_audit_log ENABLE ROW LEVEL SECURITY; CREATE POLICY "aal_owner_read" ON public.admin_audit_log FOR SELECT TO authenticated USING (public.is_platform_owner());
Garde-fous sécurité par construction — toutes les actions destructives passent par des Edge Functions qui (1) vérifient is_platform_owner(), (2) exigent un motif texte non vide, (3) écrivent dans admin_audit_log avant de muter. L'impersonation ouvre une session JWT signée avec un claim spécial act_as + exp 30 min, jamais persistée côté serveur — ainsi pas de token volé.
09

Phasing proposé · à arbitrer ensemble

Phase 1 — fondations opérationnellesà livrer
  • Flag is_platform_owner + helper SQL + RLS adapt
  • Sous-app /admin avec sidebar nuit + mode-bar or
  • Tableau de bord KPI plateforme (cabinets, utilisateurs actifs, missions, MRR placeholder)
  • Liste cabinets avec actions sûres (voir, suspendre, exporter)
  • Détail cabinet · vue d'ensemble · membres · activité
  • Recherche utilisateur + reset password / déblocage / changement rôle
  • Audit log persistant + filtres + export CSV
  • Doc opérationnelle (comment se nommer platform_owner manuellement)

Effort estimé · ~14-18 h

Phase 2 — extensionaprès retour d'usage
  • Impersonation JWT signé · session 30 min · bandeau or persistant
  • Feature flags par cabinet / utilisateur / global
  • Onboarding wizard nouveau cabinet (création org + invitation owner + plan)
  • Tickets support (intégration Linear / Notion / interne)
  • Métriques business (MRR réel, churn, NPS, requêtes IA / tokens consommés)
  • Suppression définitive avec confirmation à 2 facteurs
  • Notifications RGPD à l'utilisateur impersonné

Effort estimé · ~10-15 h

Questions ouvertes pour cadrer Phase 1
  1. Combien de platform owners ? Toi seul, ou tu prévois 2-3 personnes ?
  2. Listing cabinets : on affiche les cabinets vraiment payants ou aussi les comptes en démo / désactivés ?
  3. MRR / facturation : on intègre Stripe dès Phase 1 ou on laisse un placeholder ?
  4. Reset password : on déclenche l'email Supabase Auth standard ou on génère un magic-link interne ?
  5. Audit log retention : 12 mois, 24 mois, ou indéfini ?
